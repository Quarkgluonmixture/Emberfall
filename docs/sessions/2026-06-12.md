### art:review harness autopsy (important)

The before/after review was confidently hallucinating: with all 24 images
@-attached in ONE call, the gemini CLI silently drops them and the model
"reviews" from the brief alone. Three runs, three stories (praise for
phantom changes / opposite complaints / "zero pixel difference" + the
original audit's scores). Pixel-diff proved baseline↔current really
differed (3-4% of pixels, shoreline); a 2-attachment call described the
same pair accurately — attachment count is the failure axis.

**Fix shipped**: `art-review.mjs` now makes one gemini call per pair
(2 attachments) + a text-only synthesis; the 13-category re-score is
dropped (that's `art:audit`'s job). Prompts demand pixel-evidence-only
claims ("unchanged is a fine answer"). NOTE: `art:audit` attaches 12 images
in one call and may sit near the same cliff — restructure it the same way
next time it's load-bearing.

### art:review hardened again (blind-call detection)

Per-pair calls can ALSO go silently blind under rate pressure (two
concurrent CLI sessions + ~40 prior calls produced a review full of
phantom claims — "light tan UI", "staff citizens" — none in the images;
a single manual pair call minutes later was accurate and confirmed
`gemini-3.1-pro-preview` served). Fix: art-review now measures a pixel
diff per pair (sharp, >8/channel threshold) and cross-checks the verdict —
"unchanged" with big diff (or a change-claim with ~zero diff) retries once,
then gets a ⚠ UNRELIABLE tag the synthesis is told to ignore. Report
header counts reliable verdicts. Rule of thumb: never run art:audit and
art:review concurrently.

### art:audit restructured (same per-shot pattern as art:review)

One gemini call per screenshot (1 attachment) → grounded per-shot critiques
→ text-only synthesis producing the 13-category scores + A-H sections →
distill (unchanged). Validation run completed: critiques reference real
on-screen content. Caveat: single-image absolute critique skews much
harsher than comparative review, and individual claims can still be flat
wrong (it claimed "no action icons" on a shot that has them) — treat the
audit as a problem-finding direction, verify specifics before implementing.

### Final review round — verdict-noise calibration

Even hardened (per-pair + pixel-diff cross-check), Improved/Regressed
direction flips run-to-run on wholesale-change pairs (~50-70% pixels
differ): mid-day went Improved→Regressed, close-settlement
Improved→Regressed on identical images. Every concrete "Regressed" claim
was cross-examined against crops at the named location: mountain stripes
were the one real find (fixed); "broken rivers at Fernfell" = a
pre-existing 1-tile lake the brighter art + shore rings now spotlight;
"green pads under settlements" and "icons disappeared" don't survive
inspection. Calibration rule going forward: the review is a FINDER —
verify each named regression with a crop before acting; QA assertions +
human eyeball remain the actual gate when a pass swaps the whole palette.
